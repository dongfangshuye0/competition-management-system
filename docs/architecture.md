# 高校竞赛管理系统架构说明

## 目标

本项目提供一个基于 Spring Boot 3.2.6 + Vue3 前后端分离的高校竞赛管理系统，覆盖竞赛发布、报名审核、作品提交、教师评分、成绩排名、通知和智能答疑等流程。

## 模块职责

### 前端（`frontend/`）
- **页面组件**：学生/教师/管理员三端独立视图，按角色路由隔离
- **状态管理**：Pinia 管理用户登录态、通知未读数等全局状态
- **API 封装**：Axios 实例统一注入 JWT Token，统一错误处理
- **UI 组件**：Element Plus 提供表格、表单、弹窗等基础组件
- **路由守卫**：Vue Router 导航守卫校验角色权限，未登录跳转登录页

### 后端（`backend/competition/`）
- **Controller**：只负责 HTTP 路由和请求对象绑定，业务判断下沉到服务层
- **Service**：实现竞赛、报名、作品、评分、通知、用户等核心流程
- **Mapper**：MyBatis-Plus 实现数据库持久化
- **Security**：JWT 拦截 `/api/**` 登录态，公开接口在拦截器中排除
- **Exception**：统一业务异常和系统异常响应

### 数据层
- **MySQL**：6 张业务表（user, competition, registration, work, score, notification）
- **文件存储**：作品文件和报名表上传至本地磁盘，可通过配置切换对象存储

### 数据访问约定
- **查询下推**：看板统计、报名筛选、作品筛选、排名评分和通知未读数优先在 `DatabaseStore` 中提供按场景命名的查询或计数方法，避免业务层反复全量读取再过滤。
- **索引配套**：`registration`、`work`、`notification` 等高频查询表需要在 `sql/init.sql` 中同步维护对应索引，新旧环境通过 `information_schema` 兼容补齐。
- **业务层职责**：`PlatformService` 负责流程编排和响应组装，不直接承担查询策略优化；新增列表或统计接口时，应先复用已有数据访问方法。

## 开发流程

```bash
# 启动后端（端口 8080）
cd backend/competition
mvn spring-boot:run

# 启动前端（端口 5173，自动代理 /api → 8080）
cd frontend
npm install
npm run dev

# 生产构建
npm run build    # 输出到 frontend/dist/
```

## 关键业务流程

1. 用户登录后获得 JWT，前端 Axios 拦截器自动在请求头注入 `Authorization: Bearer <token>`
2. `AuthInterceptor` 解析 token，读取当前用户并写入 `AuthContext`
3. 学生只能在报名窗口内报名；报名审核通过后，必须等竞赛进入作品提交窗口才能上传作品
4. 教师只能审核自己指导的报名，或评审自己指导/被分配的作品；管理员拥有全局管理权限
5. 成绩排名默认隐藏，管理员公布后公开接口才返回排名
6. 提交作品后自动触发 DeepSeek AI 初评（异步或同步，视配置而定）

## 当前限制

- 当前主业务数据保存到 MySQL `competition_db`，首次启动会补齐演示数据
- 默认数据库连接为 `root/123456@localhost:3306/competition_db`，生产环境应通过 `DB_URL`、`DB_USERNAME`、`DB_PASSWORD` 覆盖
- 智能答疑和作品初评当前是本地规则模拟，尚未接入真实 DeepSeek API
- 前端为独立 Vite + Vue3 项目，与后端分开启动和部署
